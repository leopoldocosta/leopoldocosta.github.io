MTU por interface e o comportamento do MSS

As setas verticais apontam para cada interface de cada elemento, com a MTU ali configurada. Embaixo, a esteira mostra o TCP MSS sendo negociado no handshake: o cliente e o servidor anunciam MSS 1460, mas o roteador do túnel reescreve o valor (MSS clamping) para 1360, garantindo que o segmento caiba na MTU de 1400 B do túnel.

Interfaces físicas

MTU 1500 nas pontas

Cliente, firewalls, WAF, servidor e as interfaces LAN/WAN dos roteadores ficam em 1500. São quadros Ethernet padrão; ninguém precisa baixar a MTU aqui.

Interface lógica Tun0

MTU 1400 no túnel

Só a interface do túnel é reduzida: ip mtu 1400. É a única MTU diferente do caminho — e a que governa a sessão inteira, por ser a menor.

Na mesma interface Tun0

MSS clamp → 1360

O roteador reescreve o MSS dos SYN que passam: ip tcp adjust-mss 1360. Cliente e servidor passam a usar 1360 sem nem perceber.

Por que o clamp resolve sem depender de ICMP: sem ele, a sessão dependeria do Path MTU Discovery (DF=1 + ICMP tipo 3, código 4). Se o firewall ou o WAF filtrar esse ICMP, vira buraco-negro de MTU — o handshake funciona, mas transferências grandes travam. O adjust-mss atua direto no SYN, então o tamanho já chega correto às duas pontas e nada precisa ser fragmentado nem sinalizado por ICMP.
