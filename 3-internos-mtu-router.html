Internos do roteador · MTU · TCP MSS

Como um roteador tem MTU 1500
e MTU 1400 ao mesmo tempo

A MTU não pertence ao roteador — ela pertence a cada interface separadamente. O kernel do SO enxerga eth0 e tun0 como dispositivos de rede distintos, cada um com sua MTU própria. O subsistema de roteamento decide qual interface usar — e aí a MTU daquela interface passa a valer para o pacote.

Corte transversal: o que acontece dentro do roteador

O pacote "mergulha" do mundo físico (eth0 LAN, MTU 1500) para o virtual (tun0, MTU 1400), passa pela verificação, é encapsulado pelo IPsec e volta ao mundo físico (eth1 WAN, MTU 1500) como um pacote externo maior mas que ainda cabe nos 1500B.

interface física (eth) MTU 1500 interface virtual/lógica (tun0) MTU 1400 motor IPsec — encapsulamento fluxo no mundo físico fluxo no mundo virtual fluxo de volta ao físico (outer)

A conta que fecha — por que 1400 + 68 cabe em 1500

O pacote interno respeita o teto de 1400B imposto pelo tun0. O IPsec acrescenta ~68B de envelope. O total de 1468B é menor que a MTU 1500 da interface WAN — o frame cabe sem fragmentação.

Configuração por interface

Interface física

eth0 LAN · eth1 WAN

Interfaces Ethernet reais. A MTU 1500 é padrão IEEE 802.3 e não precisa ser declarada explicitamente na maioria dos sistemas.

! Cisco IOS
interface GigabitEthernet0/0
 description LAN
 ! mtu 1500  ← padrão, omissível

interface GigabitEthernet0/1
 description WAN-MPLS
 ! mtu 1500  ← padrão, omissível
Interface virtual — o ponto-chave

tun0 / Tunnel0

Interface lógica criada pelo kernel para representar o túnel. A MTU é configurada menor para acomodar os cabeçalhos de encapsulamento que serão adicionados.

! Cisco IOS
interface Tunnel0
 description VPN-IPsec
 ip mtu 1400
 ip tcp adjust-mss 1360
 tunnel source GigabitEthernet0/1
 tunnel destination <peer-ip>
 tunnel mode ipsec ipv4
O que o kernel faz internamente

Sequência na pilha IP

A pilha IP trata tun0 como qualquer outro dispositivo de rede. A MTU da interface selecionada pelo roteador define o limite para aquele pacote.

1. Pacote chega em eth0 LAN (MTU 1500)
2. ip_route_output → next-hop: tun0
3. tun0.mtu = 1400 → verificação:
   ├─ tamanho ≤ 1400B  → IPsec → eth1
   ├─ tamanho > 1400B, DF=1
   │    → ICMP type 3, code 4
   │      (Fragmentation Needed)
   └─ tamanho > 1400B, DF=0
        → fragmenta antes de encapsular
4. IPsec adiciona +68B → 1468B
5. eth1.mtu = 1500 → 1468 ≤ 1500 ✓

O que o tun0 faz com pacotes grandes

Caminho feliz — DF=1 (TCP)

Pacote ≤ 1400B: passa

O MSS clamping garante que segmentos TCP nunca excedam 1360B de dados + 40B de cabeçalho = 1400B total. Esses pacotes passam direto pelo gate do tun0 sem nenhuma fragmentação.

Exceção — DF=1 sem MSS clamp

Pacote > 1400B: ICMP de volta

O tun0 não consegue encaminhar e manda ICMP type 3 code 4 (Fragmentation Needed) de volta ao remetente. Se o firewall bloquear esse ICMP: buraco-negro de MTU.

Alternativa — DF=0 (ex.: UDP)

Pacote > 1400B: fragmenta

Se DF=0, o roteador quebra o pacote em fragmentos ≤ 1400B antes de encapsular. Cada fragmento recebe o envelope IPsec e chega separadamente. Caro em CPU; o destino remonta.

Por que o MSS clamping é a solução certa: ele age no campo MSS option do TCP SYN antes de chegar ao gate do tun0, garantindo que segmentos TCP nunca excedam 1360B de payload (+ 40B = 1400B inner). Resultado: sem fragmentação, sem depender de ICMP, funciona mesmo com firewall e WAF filtrando pacotes ICMP do tipo "Fragmentation Needed".
